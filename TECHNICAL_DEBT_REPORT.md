# Отчет об архитектурных проблемах и багах проекта Teloxide

## 1. Резюме
Проект содержит критические архитектурные уязвимости, в частности, повсеместное использование `panic!` для обработки ошибок (в том числе при работе с БД), что делает систему крайне нестабильной и уязвимой к DoS-атакам (падение при получении некорректных данных от Telegram).

## 2. Критические места (Panic & Unsafe)

| Модуль | Тип проблемы | Описание | Примеры (строки) |
| :--- | :--- | :--- | :--- |
| `src/storage.rs` | **Критично** | Полное отсутствие обработки ошибок БД, использование `panic!` | L1109, L1124, L1157, L1365-1523 |
| `src/parser/command.rs` | **Логическая ошибка** | Падение при неожиданном AST (защита от дурака) | L792, L812, L839, L866, L909 |
| `src/ingress.rs` | **DoS-вектор** | Хардкод ID, падения при ошибках диспетчеризации | L583, L632, L1187, L1577 |
| `src/moderation.rs` | **Логическая ошибка** | Падение при нештатных результатах модерации | L1596, L1618, L1669, L1689 |
| `src/parser/target.rs` | **Ошибки парсинга** | `unwrap_or_else` с `panic!` при парсинге селекторов | L218, L223, L235, L240 |

## 3. Архитектурные антипаттерны
- **God Objects:** `storage.rs`, `ingress.rs` и `moderation.rs` нарушают принцип единственной ответственности (SRP). Файлы перегружены логикой.
- **Магические числа:** В `ingress.rs` (L583, L632) используются константы `900` и `700` как fallback для `message_id`, что свидетельствует о проблемах в логике получения контекста сообщения.
- **Отсутствие обработки ошибок:** Вместо типизированных ошибок (например, через `thiserror`) используется `panic!`.

## 4. Рекомендации по исправлению (MVP Path)
1. **Error Handling:** Внедрить `thiserror` и переписать сигнатуры функций в `storage.rs` с `T` на `Result<T, Error>`.
2. **Refactoring:** Выделить из `storage.rs` модуль инициализации и миграций (`migrations.rs`).
3. **Safety:** Заменить все вызовы `panic!` в парсере на возврат `Err(ParserError::UnexpectedAST(node))`.
4. **Testing:** Добавить негативные тесты для парсера, которые вызывают ошибки, вместо использования `panic!` в самих тестах.
